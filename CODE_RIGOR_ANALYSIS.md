## 0. 先定边界：这到底是「什么级别的风险」

这是**最该先讲清楚**的一点，决定了后面所有严谨性问题的分量。

**经代码确认，本系统不是自动交易系统：**
- 全代码**无任何下单 / 执行逻辑**（`place_order` / `submit_order` / 券商 API 调用均为 0）。
- `portfolio.py` 只是**手工录入**持仓（用户自己填股数、成本价），系统不替你买卖。
- `risk_guard` 只**广播告警**到前端，不触发任何交易。
- 实时价格止损（`scan_realtime`）也是「算出来 → 推告警」，不动作。

**结论：风险边界 = 「信号 / 告警 / 报告的质量」+「是否给人错误信心」。** 系统不会自己亏你的钱，但如果你基于它给的信号、评分、回测曲线、AI 解释去做买卖决策，那么**这些输出的每一个严谨性缺口，都会直接转化为你的本金风险**。

这其实是个**相对安全的架构**（人保留最终执行权），值得肯定。但也意味着：**所有严谨性责任都压在「输出可信、不哑火、不夸大、有免责」这四点上了**——而下面逐条看，这四点都有缺口。

## 2. 回测严谨性（个人资金最危险区）

### 🔴 2.1 选股排序已退化，"策略收益"是空壳

`backtester._score_stock`（backtester.py:345-354）评分逻辑：
```python
momentum = stock.get("momentum_20d") or stock.get("change_pct") or 0
score += float(momentum)
```
- `momentum_20d` **不在因子注册表**（`factor_lib/base.py` 无此项）。
- `change_pct` 在 `stock_daily` 表，**不在 `factor_value`**；而回测查询（:52）只从 `factor_value` 取 `symbol/trade_date/factor_name/value`，所以 `day_data` 里根本没有 `change_pct` 这一列。

→ 结果：`_score_stock` 对**所有股票恒返回 0**。`candidates.sort(key=_score_stock, reverse=True)` 是稳定排序，退化为**按查询顺序取前 `top_n`**。

**危害（对个人资金极严重）**：用户在前端调因子权重、看回测曲线变化，以为在优化策略——但实际上排序根本没被因子驱动，曲线变化来自噪声/顺序，**给的是"看似精确、实则无意义"的具体收益率**。这是比"报错"更危险的失败：它给你信心，却不给你真相。

### 🔴 2.2 假数据兜底未完全受开关门控

`backtester.py` 有三处直接返回 `_mock_result`（硬编码"年化约 18%"的漂亮曲线，:503-506）：
- :121 无因子 → mock
- :128 查询无数据 → mock
- :149 无交易日 → mock

注意：**只有 `except` 块里的 mock（:243-245）受 `mock_on_error` 开关控制**。上面三处**无条件执行**，即使 `mock_on_error=False`（当前默认）也会返回假数据。

而"查询无数据"（:128）正是**因子未入库时的真实场景**——比如我们已确认的 `net_amount` 因子从未存储，用它回测就会走 :128 → 返回一条年化 18%、夏普 1.85 的假曲线，前端毫无提示。

**危害**：错误的配置 / 缺失的数据，换来一条让你放心的漂亮回测。对个人资金，这是最该杜绝的"毒药"。

### 🟠 2.3 无过拟合 / 样本外防护

- 回测在用户选定的**整段区间**上同时"选因子 + 算收益"，典型的数据窥探（data snooping）。
- 注册表有 **38 个因子**，前端可任意组合、调阈值，无 train/test 切分、无 walk-forward、无 purged embargo（防止泄漏未来信息到训练窗）、无多重比较校正。
- 报告不区分样本内 / 样本外。

**危害**：这是量化领域"回测暴赚、实盘亏钱"的头号原因。对个人投资者尤其危险——你最容易把样本内优美的曲线当成能力。

### 🟠 2.4 成本与可行性模型有缺口

- 成本**有建模**（:109-110 佣金 0.03% + 印花税 0.05%，调仓日扣除，:203），这点比很多玩具回测强。
- 但：① **无滑点**（买卖冲击 / 买卖价差）；② **无涨跌停不可成交约束**——A 股涨停买不进、跌停卖不出，日频调仓策略可行性受此影响很大；③ 印花税实际**仅卖出侧**征收（0.05%），代码在每次调仓对全仓双向扣，略高估成本。
- 日频完全调仓（turnover=1.0）下成本约 0.08%/次，年化极高——这反而是"过度惩罚"，会让真实可行的低频策略看起来很糟。成本参数也**未暴露到配置**，不可调。

### 🟠 2.5 因子名未做白名单（与选股器不一致）

`backtest.py:run_backtest` 只校验 `conditions` 非空和日期顺序（:84-87），**未把因子名与 `FACTOR_REGISTRY` 比对**。因子名直接 f-string 进 SQL（backtester.py:51-57）。

- `stock_picker.py` **已做**白名单校验，backtester **没做**——两处不一致。
- asyncpg 默认禁止多语句执行，把 SQL 注入危害降到"只会报错"，但仍应统一加白名单：既防注入，也防"无效因子名 → 静默空结果"误导用户。

### 🟢 2.6 已修好的（应保留）

- **前视偏差已修**（:177-185）：T 日因子选股、T+1 日算收益，无未来信息。
- **幸存者偏差已处理**（:280-281）：`list_date <= trade_date OR list_date IS NULL` + 退市过滤。
- **指标公式正确**：Sharpe（:400-426，无风险利率 2% 年化）、最大回撤（:428-454）计算无误。

## 3. 风控 / 本金保护严谨性

### 🔴 3.1 静默失败——对个人资金最危险的失败模式

系统在"最需要它的时候"会**悄悄哑火**，且不告警：
- `ws.py:213`：持仓列表加载失败 → 日志 `跳过实时风控检查`，**前端无任何提示**。你的实时止损告警就此消失，而你以为还在守护。
- `portfolio.py:143/179/226/255`：组合查询/增删改失败 → `使用模拟响应`（**假盈亏**）。你的资产视图可能是编的。
- `risk_guard` 各检查 `except → return [] / 跳过`（如 :287 降级为不过滤、:490 负面舆情异常跳过）。

**危害**：对一个管你本金停损的系统，"失败但安静"比"直接报错"恶劣得多。你会在毫无察觉中被套、被深套。

### 🟠 3.2 只有"价格告警"，没有"本金管理"

- 风控阈值集中在 `risk_guard.DEFAULT_RULES`（:65）且支持 API 更新——**这点做得不错，可配置**。
- 但全系统**没有任何仓位 / 本金维度的管理**：
  - 无单票仓位上限（如"单票 ≤ 总资金 10%"）
  - 无组合最大回撤熔断（如"回撤超 15% 暂停所有信号"）
  - 无行业 / 板块集中度约束
  - 无"本笔最大可亏金额"概念
- 风控 = "价格碰到你设的线，喊你一声"。它不回答最关键的问题：**这只该买多少？最多让你亏多少？**

### 🟠 3.3 两个可开启却永不开火的规则

`DEFAULT_RULES` 含 `fund_outflow`（资金异常）与 `negative_sentiment`（负面舆情）。UI 里能开启，但底层因子 `net_amount` / `sentiment_score_1d` **从未被计算 / 入库**（见第四轮评审）。

→ 用户以为"我开了资金流预警"，实际该检查永远取到 None、静默跳过。**开了等于没开，且无人告知**。

### 🟢 3.4 已修好的

- 实时价格止损已接行情流水线（`scan_realtime` 在 `ws.py` 接线）。
- 盘后技术破位（MA 跌破 / 布林破位）本轮改为从 `stock_daily` 取 `close`，能真正触发。

## 4. 数据质量与时效性

- 🟠 **行情 staleness**：QuoteGateway 轮询 akshare push2（1–3s）。若被限频或源返回过期快照，"实时价"可能是几分钟前的 → 止损基于过期价，误报或漏报。建议对行情加**时间戳新鲜度校验**（超过 N 秒标 stale 并禁用自动告警）。
- 🟠 **因子缺失连锁**：若某日因子计算失败（网络/源异常），`factor_value` 该日该因子为 None → `risk_guard` 读 None → 检查静默跳过（与 3.1 同源）。建议对"关键因子连续缺失"做告警。
- 🟢 **K 线前复权已修正**（`adjust="qfq"`），除权缺口不再造假。

## 5. AI 解释与决策误导

- 🟢 **`ai_explainer` 接地做得好**（ai_explainer.py:159-191）：解释用的因子直接从 `factor_value` 实查，`_build_explain_prompt` 只喂真实存在的因子（无数据则标"（无因子数据）"）。**不会编造不存在的因子**——这比大多数系统强。
- 🟠 **但本地 3B / 14B 模型仍可能臆测**：即使只给真实因子，小模型也可能做错误归因（"主力净流入"式脑补）。报告 `report_service` 的"原因分析"若为自由生成文本，需明确标注为"基于数据的客观描述"。
- 🔴 **全项目无"不构成投资建议"免责**：唯一相关的是 `report_service.py:295` 报告的"风险提示"小节——那只是罗列**市场**风险，不是"本系统输出不构成投资建议"的责任边界声明。对个人投资者工具，缺明确 disclaimer 既是合规隐患，也助长过度信任。

## 7. 严谨性补强清单（按对个人资金影响排序）

### P0 · 必须修（否则可能直接误导真金白银）
1. **门控 / 移除 mock 回测**：`backtester.py:121/128/149` 的 `_mock_result` 在空数据 / 无因子 / 无交易日时**必须 raise**，绝不许返回"年化 18%"假曲线（或把 `mock_on_error` 开关覆盖所有 mock 路径，并在前端红字标注"模拟数据"）。
2. **修 `_score_stock`**：用真实存在于 `factor_value` 的因子（如 `ma_20` / `rsi_14` / `macd_dif` 等 registry 因子）做排序；若设计就是"等权不分排序"，就明确去掉排序并改文案，别让用户误以为在优化 alpha。
3. **消除静默失败**：持仓加载失败 / 因子缺失时，前端显式红色横幅"⚠ 实时风控已停用 / 数据缺失"，组合盈亏失败时显示"数据不可用"而非模拟值。让失败**可见**。

### P1 · 应修（显著影响决策质量）
4. **过拟合防护**：引入 walk-forward + 保留样本外区间，报告标注"样本内 / 样本外"；对因子组合做多重比较提示。
5. **补全成本与可行性**：加滑点参数、涨跌停不可成交约束；印花税仅卖出侧；成本参数暴露到配置。
6. **本金管理框架**：单票上限 %、组合最大回撤熔断、行业集中度约束——至少给可配置项，让风控从"喊一声"升级为"管住钱"。
7. **因子名白名单**：`backtest.py` 入参按 `FACTOR_REGISTRY` 校验（与 `stock_picker` 对齐）。
8. **无效规则可见**：`fund_outflow` / `negative_sentiment` 开启但底层因子未接入时，前端明确提示"该因子未计算，规则暂不生效"。

### P2 · 建议（信任与合规）
9. **全站加"不构成投资建议"免责**，AI 解释标注为客观描述；回测页加"历史业绩不代表未来"。
10. **行情新鲜度校验** + **因子缺失告警**。
11. **回测示例去掉 `net_amount`**（backtest.py:56，该因子未存储），换真实存在的因子，避免"点默认就跑出无效/假结果"。

## 8. 给个人投资者的实操铁律（无论代码怎么改）

1. **任何信号都只是输入之一**，不是指令。永远保留人工判断与人工执行。
2. **先用小资金 / 模拟盘验证**至少 1–3 个月，再考虑放大。
3. **单笔风险预算前置**：每笔下注前先定"这笔最多亏多少"，而非"能赚多少"。
4. **永远不满仓、不单票重仓**；分散是对未知的最廉价保险。
5. **回测曲线当反证看**：能跑通的，先假设它是过拟合，直到样本外也稳。
6. **系统喊停就减，系统哑火（无告警）也要警惕**——本项目的静默失败风险要求你定期人工确认风控在线。

## 总结

架构上（非自动交易、AI 接地、前视/幸存者已修、阈值可配）这套系统是**负责任的基础**。但在「个人资金」这把尺子下，它仍有三处会**直接转化为本金风险**的硬伤：**回测排序退化 + 假数据兜底**（给你错误的"能力证明"）、**风控静默失败**（最需要它时悄悄消失）、**缺少免责与本金管理框架**（易过度信任、无仓位护栏）。

建议优先级：**P0 三件套（关 mock / 修排序 / 失败可见）> P1（过拟合防护 / 成本可行性 / 本金管理 / 白名单）**。修完 P0，这套系统才真正"不会骗你"——在这基础上，再谈帮你赚钱。
